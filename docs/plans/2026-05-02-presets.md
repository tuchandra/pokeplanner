# Plan: Housing-arrangement presets

Status: not started. Drafted 2026-05-02. Pick this back up when ready.

## Goal

A "Presets" button that one-click loads a curated housing arrangement across the five areas, then lets the user share the URL or tweak it. Replaces or augments the existing "Load example data" empty-state action.

## Why

Building 308 Pokémon into ~80 houses by hand is the long version of a planning exercise. A handful of canonical arrangements would:

- Give the user a useful starting point that highlights how the app works.
- Surface interesting configurations the user might not have thought of (e.g. "minimum number of houses where everyone is comfy").
- Pair well with the existing share-URL system — you can hand someone a URL like "Tushar's evolutionary-line layout" and they get the whole thing.

## Preset ideas

Each preset is a function `(roster: Pokemon[]) => House[]` that returns a complete planner state. Some involve real algorithmic work, some are mostly bookkeeping.

### 1. Evolutionary lines (medium effort)
- Each evolution chain → one or more houses.
- Eevee's 9-form line splits into multiple houses; pre-evolutions go in size-1/2/4 prefabs as appropriate.
- **Data gap**: the tracker has Pokédex numbers but no `evolvesFrom` relationship. Three options:
  - (a) Add an `evolvesFrom` column to the TSV. Manual but correct.
  - (b) Hardcode well-known chains in `src/data/evolutions.ts`. Compact since there are only ~80 chains in 312 entries.
  - (c) Heuristic: consecutive Pokédex numbers usually share a chain. Right ~80% of the time but breaks on cross-line numbers (regional forms, "story" Pokémon sharing a number).
- **Recommended**: (b) — hardcode. Bounded list, zero ambiguity, works offline.

### 2. Minimum housing (high effort)
- Pack the 307 housable Pokémon into the fewest houses where everyone is "comfy" (their habitat is in the house's lighting set, and they share the house's tag intersection — see `derivedHabitats` in `src/types.ts`).
- This is bin-packing with constraints — NP-hard in general. Heuristic approach:
  1. Bucket by habitat (Bright/Dark/Dry/Warm/Humid/Cool).
  2. Within each bucket, greedy: pick the Pokémon with the most-restrictive tag set first, find the buddy with the largest favorite-tag overlap, fill to 4.
  3. Fold leftover singletons into 1- and 2-slot prefabs.
- Doesn't have to be optimal — "fewer than the user would build by hand" is enough.
- **Recommended algorithm**: greedy + locally-optimal swaps until score plateaus. ~150 LOC, exposed as `recommendArrangement(roster)` in `src/lib/recommend.ts`.

### 3. By specialty (low effort)
- One 4-slot prefab per major specialty (Build, Burn, Grow, Water, etc.); story chars into a single house; legendaries skipped.
- Doesn't optimize habitat compatibility — purely thematic.
- ~30 LOC.

### 4. By area / habitat-coherent (low effort)
- WW (Bright fire & nature), BB (Cool/Humid water), RR (Dry stone & burn), SS (Bright sky/flying), PT (Bright/diverse).
- For each area, bucket Pokémon by habitat then build prefab-4 houses sharing favorites.
- Like `Compatible` in the picker but materialized into actual houses.

### 5. Litter chains (low effort)
- One house per litter item: pair Litter-specialty mons that drop the same item, plus a Gather Honey or Recycle mon for utility.
- Uses `litterDrop` field already in `Pokemon`.

### 6. Story slots (trivial)
- One Custom-4 house per area to display story characters and legendaries.
- Mostly cosmetic but useful as a complement to other presets.

## UX

**Trigger**: A "Presets" button in the topbar (between `+ House` and the theme toggle, or as a dropdown next to `+ House`).

**Surface**: clicking opens a popover with the list of presets — each row is `[icon] [Title] [short description]`. Clicking a row applies it.

**Apply behavior**: same confirmation pattern as the share-URL hydrator — if the user has any houses, prompt before overwriting. Successful apply selects no house, clears the picker filter, and switches the view to "All" so the user immediately sees everything.

**Sharing**: presets reuse the existing share-URL system. There's no special preset routing — it just produces a planner state, and the user can hit Share to get a URL.

## Design hooks already in place
- `loadExample()` in the store is the existing model — atomic state replacement. Each preset can be a sibling action: `applyPreset('evolutionary' | 'min-houses' | 'by-specialty' | …)`.
- `derivedHabitats(house, lookup)` already gives us the compatibility check we need for the minimum-housing scorer.
- `recommend()` in `src/lib/recommend.ts` has the per-Pokémon-per-house scoring that the minimum-housing algorithm wants.
- `partitionGroups(roster)` already partitions into Story / Misc / specialties — useful for the by-specialty preset.

## Suggested rollout
1. Land the Evolution data (option `b`: hardcoded `src/data/evolutions.ts`). Independent of presets.
2. Build the simplest preset (#3 by-specialty or #5 litter-chains).
3. Ship the Presets popover skeleton with that one preset wired up. Establishes the UX pattern.
4. Add #1 evolutionary lines next (uses data from step 1).
5. Add #4 by-area.
6. Save #2 minimum-housing for last — it's the most complex.

## Open questions
- Should presets be writable (user-defined / saved)? Probably not in v1; locally-saved presets via localStorage would be a separate feature.
- Should presets be linkable directly (e.g. `?preset=evolutionary` in the URL hydrating at load)? Easy add but not required for v1 — users can just apply, then Share.
- Is there a "compact" mode that shows the user the preset without committing? E.g. "preview" mode where you can see the houses without replacing your local state. Out of scope for v1.
